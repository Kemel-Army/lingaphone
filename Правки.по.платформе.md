# Lignaphone — Online-School Platform: Phased Implementation Plan

> *Reader:* This plan is written to be executed one phase per fresh chat context. Each phase is self-contained: it names the docs to read, the exact APIs allowed, what to build, how to verify, and what NOT to do. Do phases in order — later phases assume earlier ones exist.

## Tech stack (decided)

- *Framework:* Nuxt 4 (4.4.x), Vue 3, <script setup lang="ts">.
- *Server / BFF:* Nitro server routes under server/ (NO separate REST service — typed endpoints consumed directly via $fetch/useFetch). This is the BFF.
- *DB:* PostgreSQL + Drizzle ORM (drizzle-orm/node-postgres with pg Pool).
- *Auth:* BetterAuth (better-auth) with Drizzle adapter, admin + organization plugins.
- *UI:* Nuxt UI v4 (@nuxt/ui ^4.9), Tailwind CSS v4.
- *Realtime:* Nitro native WebSocket (defineWebSocketHandler) + VueUse useWebSocket. *Deploy = single always-on VPS (locked)* → in-memory Nitro WS pub/sub is sufficient; NO Supabase/Ably/Pusher needed. (Run one Node process, or add a shared broker only if scaled to multiple instances later.)
- *Avatar:* Rive (@rive-app/canvas) for character/emotes/customization; Matter.js (matter-js) for the physics playground.
- *Online lessons:* Jitsi (IFrame API; meet.jit.si first, self-host on VPS later).
- *Excel export:* exceljs.
- *Мессенджеры (WhatsApp + др.):* Wazzup24 aggregator API v3 (api.wazzup24.com) — *iframe widget in client card (locked path A)*. Covers WhatsApp, Telegram, Instagram, VK, Avito, Viber, MAX.
- *Deploy target:* single VPS (Node process + PostgreSQL + optional self-hosted Jitsi). Locked 2026-07-01.

## Standing rules (apply in EVERY phase — non-negotiable)

1. *Nuxt UI MCP is mandatory.* Before writing ANY Nuxt UI component markup, query the Nuxt UI MCP (mcp__nuxt-ui-remote__get-component, get-component-metadata, get-example, search-components) for that exact component to confirm prop names, slot names, and emit signatures. *Never invent or guess props/slots.* If a component is claimed here but the MCP disagrees, trust the MCP.
2. *CodeGraph for navigation.* .codegraph/ exists at repo root. Use codegraph explore "<question>" / codegraph node <symbol|file> (or the MCP tools if active) BEFORE grep/find/reading files to locate and understand code. Re-index after large structural changes.
3. *Impeccable for student-facing design.* The student cabinet + avatar UI must be cartoonish/playful. Use the impeccable skill to design/critique those surfaces before/while building them. Admin/teacher/director surfaces stay clean & professional (Liquid Glass shell).
4. *Docs before code.* Each phase lists doc URLs. Read them (WebFetch) before implementing; do not code APIs from memory.
5. *Verify > assume.* Every phase ends with a verification checklist. Run typecheck (pnpm typecheck), lint (pnpm lint), and the phase's specific checks before declaring done.
6. *No invented APIs.* Only use symbols confirmed in Phase 0's "Allowed APIs" or freshly verified from docs. If something you need isn't listed, fetch its docs and add it — don't guess.

### 0.1 BetterAuth (source: better-auth.com)

- Docs: [installation](https://better-auth.com/docs/installation), [nuxt integration](https://better-auth.com/docs/integrations/nuxt), [drizzle adapter](https://better-auth.com/docs/adapters/drizzle), [cli](https://better-auth.com/docs/concepts/cli), [database](https://better-auth.com/docs/concepts/database), [admin plugin](https://better-auth.com/docs/plugins/admin), [organization plugin](https://better-auth.com/docs/plugins/organization).
- *Package:* better-auth (single package; Drizzle adapter ships inside it).
- *Allowed imports (exact):*
  - import { betterAuth } from "better-auth"
  - import { drizzleAdapter } from "better-auth/adapters/drizzle" ← *NOT* @better-auth/drizzle-adapter
  - import { admin, organization } from "better-auth/plugins"
  - import { createAccessControl } from "better-auth/plugins/access"
  - import { createAuthClient } from "better-auth/vue"
  - import { adminClient, organizationClient } from "better-auth/client/plugins"
- *CLI:* npx @better-auth/cli generate (generates Drizzle schema). *migrate is NOT supported for Drizzle* — after generate, apply with Drizzle Kit.
- *Core tables:* user, session, account, verification. admin plugin adds role, banned, banReason, banExpires to user (role = comma-separated string → multi-role). organization plugin adds organization/member/invitation tables + activeOrganizationId on session; teams (opt-in) add team/teamMember.
- *Server session read:* const session = await auth.api.getSession({ headers: event.headers }).
- *Env:* BETTER_AUTH_SECRET (≥32 chars), BETTER_AUTH_URL.
- *ANTI-PATTERNS:* don't import @better-auth/drizzle-adapter; don't run better-auth migrate with Drizzle; don't set custom role from client at signup (input: false).

- *ANTI-PATTERNS:* don't use @rc/v2 relations with stable install; don't run db:push against prod; don't create a new Pool per request (singleton only).
